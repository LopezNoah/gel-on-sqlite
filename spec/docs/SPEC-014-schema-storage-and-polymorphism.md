# SPEC-014: Schema Storage and Polymorphism in PostgreSQL

Status: Draft
Owners: Schema Team and PostgreSQL Backend Team
Last Updated: 2026-04-02

## Purpose

Define how Gel schema objects are represented in PostgreSQL-backed storage, how object/link/property data is laid out physically, and how abstract or polymorphic object semantics are compiled into executable SQL.

This spec is intended to be the canonical reference for migration and backend work that must preserve physical layout compatibility and polymorphic query behavior.

## Scope

- Physical storage model for object types, links, properties, and link properties in PostgreSQL.
- Catalog-level metadata sources used to map schema pointers to backend table/column locations.
- Behavior for abstract/non-concrete types and polymorphic reads/dispatch.
- Query-shape implications for inheritance views and union-based polymorphic plans.

## Non-Goals

- SQLite backend storage details and migration format (handled in separate backend adaptation specs).
- Full user-facing SDL tutorial for schema declaration syntax.
- PostgreSQL planner internals beyond the SQL artifacts generated by Gel.

## Requirements

- R1: Every concrete object type with storage must map deterministically to a backend table identity.
- R2: Pointer storage placement (inline object column vs pointer/link table) must be deterministic from cardinality and pointer properties.
- R3: Object references in backend row storage must use UUID-compatible representation.
- R4: Link properties must resolve to stable backend columns in link tables, including special handling for `source` and `target`.
- R5: Abstract/non-concrete inheritance reads must expose a unified relation over participating concrete descendants.
- R6: Polymorphic reads over inheritance must preserve column compatibility across union arms.
- R7: Type dispatch for object-parameter polymorphic function overloads must select the nearest matching ancestor implementation.
- R8: Introspection layers must map physical UUID column names to schema pointer identity and human-facing names.
- R9: Missing or unsupported storage forms must fail explicitly with diagnostics instead of silently degrading semantics.
- R10: Migrations that move pointer storage between layouts must preserve value semantics during data movement.

## Physical Model

- **Schema object IDs**
  - Schema metadata uses UUID identifiers (for example `_SchemaObjectType.id`, `_SchemaPointer.id`, `_SchemaLink.id`) in PostgreSQL-side metaschema representation.
  - UUID identity is used as a stable bridge between logical schema objects and backend artifacts.

- **Object type rows**
  - Concrete object types are stored in object tables when `has_table(...)` returns true.
  - Inline pointer columns on object tables use pointer UUID text as column names for most user pointers; special pointers such as `id` and internal names may preserve symbolic names.

- **Pointer/link table rows**
  - Multi pointers or pointers with user-defined properties are stored in dedicated pointer/link tables.
  - Link tables carry `source` and `target`; link properties are stored on link tables, using pointer UUID-derived column names except special `source`/`target` cases.

- **Object references and UUIDs**
  - Object-typed pointer storage resolves to PostgreSQL `uuid` columns.
  - Pseudo/internal references (for example certain scalar pseudo-links) may not materialize to physical tables and are handled by compiler rules.

## Worked Example (Feature-Complete)

### Example EdgeQL Schema (illustrative)

```edgeql
module default {
  abstract type Content {
    required title: str;
    multi tags: str;
  }

  type User {
    required name: str;
    multi authored -> Content {
      created_at: datetime;
      role: str;
    }
  }

  type Post extending Content {
    body: str;
    required author -> User;
    multi comments -> Comment;
  }

  type Comment extending Content {
    body: str;
    required author -> User;
    required post -> Post;
  }
}
```

### Visual Storage Model (conceptual)

```text
default::Content (abstract)
  |
  +-- default::Post        -> table: <post_table_uuid_or_name>
  |      inline columns: id, __type__, title, body, author(pointer UUID col)
  |      multi link table: post.comments(source, target)
  |
  +-- default::Comment     -> table: <comment_table_uuid_or_name>
         inline columns: id, __type__, title, body, author(pointer UUID col), post(pointer UUID col)

  inline columns: id, __type__, name
  multi link table: user.authored(source, target, created_at(linkprop UUID col), role(linkprop UUID col))

default::Content.tags      -> pointer table for multi property(source, target)
```

### Abstract/Polymorphic Read Example

| Query intent | Effective SQL shape | Notes |
|---|---|---|
| Read `Content` polymorphically | `SELECT ... FROM Post_table UNION ALL SELECT ... FROM Comment_table` | inheritance view composes base/descendants via `UNION ALL` |
| Read link targets across descendants | union of link projections into canonical `(source, target)` | used in migration/policy flows for multi-link operations |
| Resolve overload for object-param function | ancestry walk + first match (`ORDER BY index LIMIT 1`) | set-returning overload arms are merged by `UNION ALL` |

## Behavior and Flows

- **Pointer storage resolution flow**
  - Resolve material pointer if needed.
  - Determine whether pointer is link property, singular, multi, or has user-defined properties.
  - Choose storage site:
    - source object table column when storable inline,
    - pointer/link table when multi or property-bearing,
    - or explicit no-storage path for unsupported/missing cases.
  - Resolve backend column SQL type from schema/IR target type.

- **Abstract and polymorphic read flow**
  - Build an inheritance view for an abstract ancestor (or any queried type with descendants in scope).
  - Include base plus concrete descendants with compatible projected columns.
  - Emit SQL as a `UNION ALL` chain over component `SELECT`s, preserving expected path/shape columns.

- **Polymorphic function overload dispatch**
  - Build candidate overload arms by object parameter type.
  - Derive nearest matching implementation using `_object_ancestors` and ordered ancestor selection.
  - For set-returning overload dispatch, emit arm composition via `UNION ALL`.

- **Introspection name reconstruction flow**
  - For UUID-named physical columns, map through pointer UUID to schema pointer metadata.
  - For special symbolic columns (`id`, `source`, `target`), map directly.
  - Expose stable SQL-facing views that resolve backend UUID identifiers into user-facing names.

- **Migration/storage-transition flow**
  - When pointer storage mode changes (object table <-> pointer table), execute explicit data backfill/update SQL.
  - Keep schema and data in lockstep with deterministic DDL + DML sequencing.
  - Reject incompatible transitions where semantics cannot be preserved.

## Verified Implementation Notes

- Inheritance views for source/pointer hierarchies are assembled as `UNION ALL` chains (`edb/pgsql/inheritance.py:99`, `edb/pgsql/inheritance.py:107`).
- Object-parameter polymorphic function compilation uses ancestor matching and `UNION ALL` composition for set-returning overload arms (`edb/pgsql/delta.py:1334`, `edb/pgsql/delta.py:1359`).
- Pointer storage decision logic (inline vs link table), link-property normalization, and UUID object reference typing are implemented in `edb/pgsql/types.py` (`edb/pgsql/types.py:561`, `edb/pgsql/types.py:594`, `edb/pgsql/types.py:599`, `edb/pgsql/types.py:755`).
- Introspection compatibility logic that maps UUID-named backend columns to schema pointer identity appears in `edb/pgsql/metaschema.py` (`edb/pgsql/metaschema.py:7092`, `edb/pgsql/metaschema.py:7093`, `edb/pgsql/metaschema.py:7094`).
